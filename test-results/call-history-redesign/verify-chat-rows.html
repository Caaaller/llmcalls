Top issues (prioritized)

1. No visual separation between rows — bubbles are invisible until hovered

The spec says bubbles should have no background normally, only on hover/active. But the only thing distinguishing one row from the next is margin-bottom: 6px. With no background, no border, no avatar, no divider, and meta text in 9–10px gray, the rows visually mush together — which is exactly what the user means by "there's no background, it looks bad".

Fix: give the bubble a faint resting background (e.g. background:#fafafa for left/right, transparent for center) OR a 1px subtle border (1px solid #eef0f3), OR an avatar/icon column on the left/right. Pure naked text on white reads as a broken layout.

2. max-width: 78% is too wide — bubbles from opposite sides collide and the chat metaphor is lost

At 78% max-width, a left bubble and right bubble in adjacent rows will overlap horizontally in the reader's eye, defeating the "left vs right = AI vs user" cue. Real chat UIs cap around 70% and most use 65%.

Fix: max-width: 70% (or 65%). Also add min-width: 0 to prevent flex children from refusing to shrink on long lines.

3. Border-radius 4px is wrong for chat bubbles

4px is a "card corner". Chat bubbles read as bubbles at 12–16px, ideally with one corner squared toward the speaker (e.g. left bubble: border-radius: 14px 14px 14px 4px). At 4px the rows look like generic list rows, not bubbles — which contributes to "looks bad".

Fix: bump to 12–14px and asymmetrically square the speaker-side corner per side.

4. Active state uses !important + inset box-shadow that will clash with the new resting background

.event-row-active .event-bubble { background: #eef1ff !important; box-shadow: inset 0 0 0 1px #c5cdf5; } — the !important is a code smell (likely added because hover was overriding it). Once you add a resting background (issue #1), drop !important and order the selectors properly: resting → hover → active.

Fix: remove !important, ensure CSS order is .event-bubble → :hover → .event-row-active.

5. Center events (System / Transfer / End / Hold) have no visual treatment to distinguish them as system events

Center-aligned bubbles get the same naked styling as left/right ones. In real chat UIs, system events are usually a small pill, full-width thin divider, or italic muted text — never a left-aligned chat bubble that happens to be centered. Right now "Call terminated" looks identical to a user message, just centered.

Fix: for .event-side-center .event-bubble, use a pill style: small uppercase muted text with a horizontal rule on each side, or a tiny rounded chip (background:#f3f4f6; color:#6b7280; font-size:11px; padding:2px 10px; border-radius:999px;). Hide the type badge for center events — it's redundant with the centered placement.

6. The type badge is duplicative information and visually noisy

Every row already conveys author via left/right placement. Adding a colored "USER" / "AI" / "DTMF" chip on top means you're saying the same thing twice, and the chips are the only colored elements on the page so they dominate visually. The user said "only the type badge chip should be colored" — that's the spec, but in practice with naked bubbles the chips become the loudest thing on screen and look like floating tags.

Fix options: (a) drop the badge entirely for USER/AI/DTMF/IVR since side placement covers it, keep it only for less obvious types (Info Req/Res, Transfer); (b) keep all badges but desaturate to a single muted gray for non-semantic ones; (c) move the badge inline with the time as a tiny uppercase label rather than a colored pill.

7. Meta row (badge + time) adds visible whitespace above every message — wastes vertical space

.event-meta { margin-bottom: 2px } plus 9px badge + 10px time on its own line means each row is ~20px of meta + ~18px of text + 6px gap ≈ 44px per row. For a long call this scrolls forever. Real chat UIs put time on hover or only show it when sender/minute changes.

Fix: hide .event-time by default, show on row hover; or only show time on the first message of each minute. Put the badge inline before the text on the same line.

8. Click target is invisible — no affordance that the row is seekable

cursor: pointer only fires once the mouse is over the row. Without a resting background, the user doesn't know the row is clickable until they're already on it. Combined with issue #1, this is why the panel "looks bad" — it looks like static text.

Fix: paired with issue #1's resting background, also add a subtle hover lift (slightly stronger bg) and consider an icon (▶ or ⏱) that appears on hover indicating "seek to this moment".
